# P4 Product Experience Acceptance

P4 turns the existing muxdev governance stack into a lower-friction product
surface. The goal is not to remove approvals, evidence, memory, or rollback; it
is to make those controls visible and approachable from the first run.

## Delivered Surface

- One-line install path: `pipx install muxdev`, with `uv tool install muxdev`
  and editable repo install documented as alternatives.
- Short quickstart: `muxdev setup --project`, `muxdev provider setup`,
  `muxdev demo --mock`, then `muxdev`.
- Provider setup wizard: `muxdev provider setup` and
  `/api/product/experience` expose install, account, and doctor steps.
- Project context anchor: `MUXDEV.md` is generated by project setup and can be
  created with `muxdev context --write`.
- Mission Control product surface: Dashboard includes Product Experience,
  budget, provider health, Git safety, rules, skills, Kanban task management,
  and optional web/IDE extension API hints.
- Git-native safety: the product surface points users to diff, dry-run ship,
  rollback, Git restore/revert, and reviewed Git commit commands without
  executing destructive actions.
- Rules and skills visibility: active profile, gate, role bindings, discovered
  skills, and management commands are exposed in one payload.
- Cost visibility: active tasks, total cost, token usage, high-cost task count,
  and budget controls are shown in API/Dashboard/CLI summaries.

## Verification

- `tests/test_p4_product_experience.py` covers setup-generated `MUXDEV.md`,
  the product experience payload, CLI provider/context/experience commands, and
  API/Dashboard exposure.
- Existing Mission Control tests continue to cover Kanban, Action Center,
  provider action wizard, approval review, provider health, and setup status.

## User-Facing Commands

```powershell
muxdev setup --project
muxdev provider setup
muxdev context --write
muxdev experience
muxdev dashboard
```
